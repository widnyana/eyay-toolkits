# BMad Sprint Runner

Running a sprint by hand means context-switching between stories, babysitting quality gates, handling retries, and making judgment calls on failures — for hours, across dozens of stories. That's the job the sprint runner takes over.

It drives Claude Code through an entire sprint: creates stories, implements them, runs typecheck and tests, invokes code review, and keeps going until everything is `done` or `blocked`. You can close the terminal and come back to results.

Two ways to run it:

- **Skill mode** — `/bmad-sprint-run` inside Claude Code. Claude orchestrates the loop itself.
- **Companion mode** — `sprint-runner.py` as a standalone Python process. It manages Claude Code as a subprocess, owns all state transitions, and enforces process-level constraints (budget, retry limits, locking) that Claude can't enforce on itself.

For unattended runs, use companion mode. The skill is useful for interactive supervision or when you don't want to set up the Python script.

---

## What it needs before it can run

- Claude Code CLI installed and authenticated (`claude --version`)
- A feature branch — it refuses to run on `main`/`master`
- `docs/_bmad_output/implementation-artifacts/sprint-status.yaml` generated by `/bmad-sprint-planning`
- BMad skills installed: `bmad-create-story`, `bmad-dev-story`, `bmad-code-review`
- Python 3.10+ (companion mode only)
- `ruamel.yaml` optional, but recommended — without it, YAML writes strip comments

```bash
pip install ruamel.yaml
```

## Quick start

```bash
cd /path/to/your/project

# See what it would do next without touching anything
python3 /path/to/bmad-sprint-run/sprint-runner.py --dry-run

# Run everything
python3 /path/to/bmad-sprint-run/sprint-runner.py

# Watch it live in a second terminal
tail -f sprint-logs/sprint-summary.log
```

## CLI reference

```
python3 sprint-runner.py [OPTIONS]
```

| Flag | Default | What it does |
|------|---------|--------------|
| `--dry-run` | — | Resolves next action and prints it, no Claude invocations |
| `--story STORY` | — | Run one story only (e.g. `4-1` or `4-1-fan-wallet-masking`) |
| `--epic EPIC` | — | Run all stories in one epic (e.g. `4`) |
| `--retry-budget INT` | 3 | Max dev+review cycles per story before blocking |
| `--skip-code-review` | — | Skip the code review step entirely |
| `--effort LEVEL` | high | Claude effort: `low`, `medium`, `high`, `xhigh`, `max` |
| `--model MODEL` | — | Override model (e.g. `sonnet`, `opus`) |
| `--log-dir PATH` | ./sprint-logs | Where to write log files |
| `--watch` | — | Tail this project's session logs and runner state (run in a second terminal) |
| `--claude-path PATH` | claude | Path to the Claude CLI binary |
| `--allowed-tools TOOLS` | — | Comma-separated list of allowed tools |
| `--debug` | — | Show sprint-runner internal debug output on console |
| `--sprint-status-path PATH` | auto-located | Override sprint-status.yaml path (otherwise auto-discovered) |
| `--story-dir PATH` | auto-located | Override story files directory |
| `--runner-state-path PATH` | auto-located | Override runner state file path |
| `--digest-path PATH` | auto-located | Override context digest path |

### Locating sprint-status.yaml

The runner finds `sprint-status.yaml` on its own, so it can be launched from any
subdirectory of the project. It walks up from the current directory looking for
the conventional `docs/_bmad_output/implementation-artifacts/sprint-status.yaml`,
then falls back to a bounded search for any `sprint-status.yaml` under a
`_bmad_output` directory. The story-dir, runner-state, and digest paths default
to the directory the discovered file lives in.

Pass `--sprint-status-path` to point at a non-standard location explicitly —
that disables discovery and the sibling paths fall back to their own defaults
unless also overridden.

### Environment variables

CLI flags take precedence over env vars.

| Variable | Flag |
|----------|------|
| `SPRINT_RETRY_BUDGET` | `--retry-budget` |
| `SPRINT_CLAUDE_PATH` | `--claude-path` |
| `SPRINT_LOG_DIR` | `--log-dir` |

## How a story moves through the runner

```
backlog
  → create-story (bmad-create-story)
  → ready-for-dev
  → in-progress (bmad-dev-story)
  → review
  → quality gate (typecheck + tests)
  → code review (bmad-code-review)
  → done

any step → retry (up to --retry-budget times) → blocked
```

A story is `done` when all three gates pass: typecheck exits 0, tests exit 0, and code review has zero critical findings. If a story needs human judgment (`decision_needed` in review), it gets `blocked` immediately — the runner doesn't guess.

Workspace detection is automatic. The runner reads `CLAUDE.md` for build commands, then scans for `package.json`, `Cargo.toml`, `pyproject.toml`, and `go.mod`. It maps changed files to workspaces and only runs gates for workspaces the story actually touched.

## State files

Everything that survives process restart lives in `docs/_bmad_output/implementation-artifacts/`:

| File | What it tracks |
|------|----------------|
| `sprint-status.yaml` | Every story's status |
| `.sprint-runner-state.yaml` | Current story, checkpoint hash, retry count, phase, blocked list |
| `.sprint-context-digest.md` | Rolling summary of the last 5 completed stories |

The runner re-reads all three files from disk at the start of every loop iteration. Retry counts are persisted and restored on restart — a story that failed twice before a crash picks up with one retry left, not three.

---

## Crash recovery and restarts

If the runner stops (Ctrl+C, crash, OOM), re-run the same command. It picks up at the right state:

- Story was `in-progress`: the dev cycle re-runs, keeping any commits already made
- Story was `review`: quality gate and code review re-run
- Retry count: restored from `.sprint-runner-state.yaml`

The runner never rolls back. It does not run `git reset --hard`, `git checkout`, `git restore`, `git clean`, or `rm` — `git_ops.py` is read-only by construction. A failed story retries by fixing forward on top of its own commits; the failure detail is fed into the retry prompt. A story that exhausts its retry budget is marked `blocked` with all its commits intact, so the work is always inspectable and salvageable.

## Concurrent instances

Running two sprint runners against the same project simultaneously corrupts state — both pick the same next story, race on the same YAML files, and create conflicting commits.

The runner prevents this with a kernel-level file lock (`fcntl.flock` on `.sprint-runner.lock` at project root). The second instance fails immediately with the PID of the running one:

```
ERROR: Another sprint-runner is already running (PID 12345).
If that process is dead, remove .sprint-runner.lock and retry.
```

The lock is released automatically by the kernel — even on `kill -9` or OOM kill. If the lock file remains after a crash (stale), the next run acquires it without intervention.

Claude Code sessions running the skill directly (not via the Python companion) do an advisory check only: they read the PID from the lock file and check if the process is alive, but cannot hold a kernel lock across tool calls. For unattended runs, always use the Python companion.

---

## Process safety

When the runner is killed, it terminates the Claude Code subprocess before exiting — Ctrl+C, `kill`, or `kill -9`. The mechanism:

- **SIGINT / SIGTERM**: caught by the runner's signal handler, which calls `proc.terminate()` then `proc.wait()` on the Claude Code process.
- **SIGKILL (OOM)**: the runner can't catch this, but Claude Code is started with `PR_SET_PDEATHSIG=SIGTERM` (Linux). When the parent process dies, the kernel delivers SIGTERM to Claude Code automatically.

## Logging

Three outputs, always written:

- `sprint-logs/sprint-run-YYYYMMDD-HHMMSS.log` — full debug log for this run
- `sprint-logs/sprint-summary.log` — one line per lifecycle event across all runs
- Console — INFO level, colored

The `--watch` flag adds a fourth: it tails the Claude Code session JSONL file in real time and streams formatted output (assistant text, tool names, turn duration, cost) to stdout while each invocation runs. Useful when you want to watch what Claude is actually doing.

## What gets injected into every Claude Code invocation

Every subprocess receives this as an appended system prompt:

```
Quality Standards (NON-NEGOTIABLE):
- Do NOT cut corners or take shortcuts
- Do NOT find easy fixes — find correct fixes
- Do NOT chase easy wins — implement features properly and completely
- Follow high-quality software engineering standards at all times
- Every function must work as specified — start it, finish it
- No partial features, no TODO stubs for core logic
```

Plus sprint context: story key, retry attempt number, recent sprint digest, and previous failure details on retries.

## Typical workflow

```bash
# 1. Plan the sprint (generates sprint-status.yaml)
/bmad-sprint-planning

# 2. Feature branch
git checkout -b sprint/epic-5

# 3. Start infrastructure (docker, validators, etc.)
docker compose up -d

# 4. Dry run to verify
python3 sprint-runner.py --dry-run

# 5. Run
python3 sprint-runner.py

# 6. Monitor in another terminal
tail -f sprint-logs/sprint-summary.log

# 7. Handle blocked stories manually
cat docs/_bmad_output/implementation-artifacts/.sprint-runner-state.yaml

# 8. Merge when satisfied
```

## Known limits

- **Stories run strictly sequentially.** There's no parallel story execution — the SKILL.md spec explicitly forbids it, and the state model doesn't support it.
- **Workspace discovery is heuristic.** If your project has unusual structure, commands might not be detected correctly. Override via `CLAUDE.md` entries or explicit flags.
- **Cost tracking is informational only.** The session total shown in the summary and per-invocation log is whatever Claude Code reports in stream-json — $0.00 for subscription accounts. No budget enforcement exists; the runner runs until all stories are resolved or you stop it.
- **Skill mode has no kernel lock.** Two people running `/bmad-sprint-run` simultaneously in different Claude Code sessions will corrupt state. Use the Python companion for anything unattended.
